## Responsive behaviour

`<post-sidenavigation>` adapts to the viewport:

- On **desktop**, the navigation is rendered inline, directly in the page flow. No wrapper is added.
- On **mobile and tablet**, the navigation is wrapped in a modal `<dialog>` that covers the page. It is opened with the `<post-sidenavigation-trigger>` button and closed with the close button, the `Esc` key or a click outside the dialog.

Narrow the canvas (or use the viewport toolbar) to switch between the two layouts. When the viewport goes from mobile to desktop and back, the dialog reopens closed.

## `<post-sidenavigation-trigger>`

The trigger is a render-less component that wraps a regular `<button>`. Place it in the header and link it to the navigation with the `for` attribute, matching the `id` of the `<post-sidenavigation>`. It is hidden on desktop, where the navigation is always visible, and keeps its `aria-controls` and `aria-expanded` attributes in sync with the navigation state.

## Usage in the header

On mobile and tablet, place the trigger among the header's controls; the navigation itself lives in the page body.

## Navigation content

The navigation content is authored by you and projected through the default slot. It must be a `<nav>` landmark, with a heading (`<h*>`, or a `<p>` if no heading level fits) linked via `aria-labelledby`, and `<ul>`/`<li>` lists for the items. Nested levels are nested `<ul>`s.

## Variants

Each item can optionally contain a `<post-icon>`. If at least one item on a level has an icon, the items without one are indented to align with it.

**Important:** for usability reasons, either all or none of the level-1 elements on a given level should have an icon. Never mix icon and non-icon items on the same level.

## Active item

Mark the link of the current page with `aria-current="page"`. The component highlights it as the active item, and screen reader users hear it announced as the current page. Set it from your routing, not on click.

## Accessibility

### Navigation landmark

The navigation you project is a `<nav>` landmark. Link it to its heading with `aria-labelledby` (see the examples above) so screen reader users can identify it and jump to it with landmark shortcuts.

**Why labeling matters:** if a page has several navigation regions (main navigation, breadcrumbs, this side navigation), a descriptive heading lets screen reader users tell them apart and move to the right one.

### Links and disclosure buttons

Use `<a>` for links and a `<button>` inside `<post-collapsible-trigger>` for collapsible sections. A link navigates, a button discloses; never nest one inside the other.

### Focus management

On mobile the navigation opens in a modal dialog: the rest of the page is inert, focus moves into the navigation on open and returns to the trigger on close.
