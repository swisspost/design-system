Conditionally renders content based on the user's KLP authentication state.

The `<post-login-widget>` calls the KLP session API on mount and renders either the `authenticated`
or `unauthenticated` slot depending on the response.
Both slots are fully customisable — no markup is imposed by the component itself.

## Authenticated

When the session API confirms the user is logged in, the `authenticated` slot is rendered.
The [user menu from the header documentation](https://swisspost-design-system-next.netlify.app/?path=/docs/27a2e64d-55ba-492d-ab79-5f7c5e818498--docs#user-menu) is the recommended content for this slot.

## Refreshing the state

Call the public `refresh()` method to re-fetch the session state programmatically — for example
after the user logs out via the user menu:

```js
const widget = document.querySelector('post-login-widget');
await widget.refresh();
```

## Reacting to state changes

The component emits a `post-login-change` event whenever the authentication state changes.

```js
document.querySelector('post-login-widget').addEventListener('post-login-change', e => {
  console.log(e.detail.authenticated); // true | false
});
```

## Fallback behaviour

If the session API request fails for any reason (network error or non-2xx response), the component
falls back to rendering the `unauthenticated` slot. No error is thrown.

## Accessibility

### Focus management

When the authentication state changes, the component updates the rendered slot.
Ensure that focus is managed appropriately in your application — for example, after a user logs out
and the unauthenticated slot becomes visible, move focus to the login link.

### ARIA labels

The `unauthenticated` slot content should have a descriptive accessible label.
When using only an icon with text hidden via `visually-hidden`, ensure the visible text or
`aria-label` clearly communicates the action to screen reader users.
