# Implementation Plan: Autocomplete (Issue #7129)

Branch: `feat/autocomplete-7129`

Goal: Implement a Stencil-based Autocomplete composed of `post-autocomplete`, `post-listbox`, and `post-option`, using `post-popovercontainer` for dropdown positioning, with keyboard + SR behavior per issue description.

Decisions (from issue + user input)
- Use `aria-owns` to associate combobox -> listbox.
- React integration tests: skip (no React test infra in repo; consistent with current state).
- Live region + `aria-describedby`: assume most likely approach is required for broad SR support: create a light-DOM status element (injected/managed by `post-autocomplete`) so `aria-describedby` works without crossing shadow boundaries.
- Angular forms: follow repo recommendation; rely on slotted native `<input>` (no custom ControlValueAccessor initially).


## Out of scope (per issue)
Do NOT implement these features - they may be added later:
- Virtual scrolling (for large option lists)
- Freeform text (allowing values not in options)
- Listbox header/footer slots
- Multiselect (listbox/option)
- Listbox grouping (`<post-optgroup>` or similar)
- Rich option content cloning to input
- Separate `value` attribute behavior (where displayed text differs from submitted value)

Props
- [x] `filter-threshold: number` default `0`
- [x] `clearable: boolean` default `false`
- [x] `options: string` optional idref to external `post-listbox` when not nested

Slot contract
- [x] Default slot contains the input + label/hints/messages (+ optionally a nested `<post-listbox>`).
- [x] Document that all children must be direct children of `post-autocomplete` (for `:slotted()` styling parity).

DOM discovery
- [x] Find exactly one slotted `<input>` (warn if none or multiple)
- [x] Find associated listbox:
  - [x] Prefer nested `<post-listbox>`
  - [x] Else use `options` idref to resolve external `post-listbox`

ARIA (on slotted input)
- [x] `role="combobox"`
- [x] `aria-autocomplete="list"`
- [x] `aria-expanded="true/false"` (sync with open state)
- [x] Use `aria-owns="<listbox-id>"` (decision 1)
- [x] Maintain `aria-activedescendant` to currently active option id when open
- [x] Maintain `aria-describedby` to include live region id (status announcements)

Live region (decision: light DOM)
- [x] Create/inject a hidden status element adjacent to the input (light DOM), e.g. `<span class="visually-hidden" aria-live="polite" aria-atomic="true">`.
- [x] Ensure it has an `id` that can be referenced by input's `aria-describedby`.
- [x] Update text when visible option count changes (e.g. "3 options available" / "No options available").

Events
- [x] `postFilterRequest` (cancellable) fired when `input.value.length >= filter-threshold`
  - [x] `event.detail`: `input.value`
  - [x] If not `defaultPrevented`, call `listbox.filter(input.value)`

Keyboard interactions (focus remains on input)
- [x] ArrowDown/ArrowUp: open if closed; move active option; wrap at boundaries
- [x] Alt+ArrowDown: open if closed
- [x] Enter: select active option
- [x] Escape: close
- [x] Tab: accept selection and close (do not trap tab)
- [x] Home/End: when open, move active to first/last visible option
- [x] Space: open listbox
- [x] Left/Right: do not override

Mouse interactions
- [x] Click on option selects it
- [x] Click outside closes

Selection + validation behaviors
- [x] Listen to `postOptionSelected` from listbox/options
  - [x] Update input value to selected option text
  - [x] Track current selection (store option id/value/text)
- [x] On blur:
  - [x] If no value selected, clear input
  - [x] If selection exists, restore input value to selected option text

Clearable
- [x] If `clearable`, render a clear button (inside shadow) and implement:
  - [x] clears input value
  - [x] clears selection state in listbox/options
  - [x] closes listbox
